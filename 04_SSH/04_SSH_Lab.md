## 🎯 Мета роботи

Налаштувати повний ланцюг безпарольного SSH-доступу до VM: згенерувати ключову пару, встановити публічний ключ для користувача `openclaw`, створити зручний псевдонім у SSH-конфігу та, нарешті, виконати обіцянку з Теми 3 — відкрити прихований порт `127.0.0.1:18789` у браузері через SSH-тунель, не змінюючи жодних мережевих налаштувань VM.

> **Контекст:** У Темі 3 ми сховали сервіс за адресою `127.0.0.1:18789` і пообіцяли, що в Темі 4 навчимось безпечно до нього дістатись. Сьогодні виконуємо цю обіцянку.

## 🛠 Покрокова інструкція

### Крок 1: Перевіряємо, чи є вже SSH-ключ

Перш ніж генерувати новий ключ — перевіримо, чи є вже існуючий. Зайвих ключів краще не плодити.

Виконайте на **хості** (вашому ноутбуці, не в VM):

```bash
# Перевірити вміст директорії ~/.ssh/
ls -la ~/.ssh/
```

**Очікуваний результат:**

- Якщо бачите файли `id_ed25519` та `id_ed25519.pub` — ключ вже є. Переходьте одразу до Кроку 2.
- Якщо директорія порожня або файлів немає — генеруємо нову пару:

```bash
# Генеруємо ключову пару Ed25519
# Замініть email на свій — це лише коментар для ідентифікації
ssh-keygen -t ed25519 -C "your_email@example.com"
```

На питання «куди зберегти» — просто натисніть **Enter** (стандартний шлях `~/.ssh/id_ed25519`).
На питання «passphrase» — для навчального середовища можна натиснути **Enter** (порожній пароль).

Після цього перевірте результат:

```bash
ls -la ~/.ssh/
# Мають з'явитись два файли: id_ed25519 та id_ed25519.pub
```

**Очікуваний результат:** Два файли. `id_ed25519` — це ваш секрет (600), `id_ed25519.pub` — це те, що можна розповсюджувати.

---

### Крок 2: Переглядаємо публічний ключ

Щоб «встановити замок» на сервер, потрібно знати, який він. Переглянемо публічний ключ:

```bash
# Вивести вміст публічного ключа
cat ~/.ssh/id_ed25519.pub
```

**Очікуваний результат:**

```text
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... your_email@example.com
```

Один довгий рядок. Це і є «замок», який ми зараз встановимо на VM.

---

### Крок 3: Підготовка — даємо `openclaw` можливість увійти

Коли у Темі 2 ми створили `openclaw` як системного користувача, він отримав shell `/usr/sbin/nologin` — це захист від інтерактивного входу для сервісних акаунтів. Для налагодження та роботи з VS Code Remote нам потрібен справжній bash.

Підключіться до VM через vagrant:

```bash
# На хості:
cd ~/devops-ai-assistant/
vagrant up
vagrant ssh
```

Тепер перевіримо поточний стан:

```bash
# На VM, від імені vagrant:
getent passwd openclaw
```

**Очікуваний результат:**

```text
openclaw:x:998:998:OpenClaw AI Agent:/home/openclaw:/usr/sbin/nologin
```

Останнє поле — `/usr/sbin/nologin`. Змінюємо на `/bin/bash`:

```bash
sudo chsh -s /bin/bash openclaw

# Перевіряємо:
getent passwd openclaw
# Останнє поле тепер: /bin/bash
```

**Очікуваний результат:** Рядок тепер закінчується на `/bin/bash`.

---

### Крок 4: Встановлюємо публічний ключ для `openclaw`

Ми використовуємо класичний DevOps-патерн **bootstrap через довірений канал**: у нас вже є підключення через `vagrant`, який має `sudo`. Саме через нього ми «підкладемо» ключ для `openclaw`.

Залишаючись у `vagrant ssh`, виконайте команду нижче — замінивши `ВСТАВТЕ_ВАШ_ПУБЛІЧНИЙ_КЛЮЧ` на вивід з Кроку 2 (весь рядок цілком):

```bash
# Створюємо директорію .ssh для openclaw
sudo mkdir -p /home/openclaw/.ssh

# Додаємо публічний ключ до authorized_keys
echo "ВСТАВТЕ_ВАШ_ПУБЛІЧНИЙ_КЛЮЧ" | sudo tee /home/openclaw/.ssh/authorized_keys

# Встановлюємо правильні права та власника
sudo chmod 700 /home/openclaw/.ssh
sudo chmod 600 /home/openclaw/.ssh/authorized_keys
sudo chown -R openclaw:openclaw /home/openclaw/.ssh
```

Перевіримо, що ключ записано правильно:

```bash
sudo cat /home/openclaw/.ssh/authorized_keys
```

**Очікуваний результат:** Ваш публічний ключ — той самий рядок, що ви бачили в Кроці 2.

Виходимо з VM:

```bash
exit
```

### Крок 5: Підключаємось як `openclaw` без пароля

Тест — чи все налаштовано. Запускаємо SSH напряму, без vagrant:

```bash
# На хості:
ssh openclaw@192.168.56.10
```

**Якщо бачите цей запит — це нормально (лише при першому підключенні):**

```text
The authenticity of host '192.168.56.10 (192.168.56.10)' can't be established.
ED25519 key fingerprint is SHA256:...
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

Введіть `yes`. SSH збереже відбиток сервера у `~/.ssh/known_hosts` — при наступних підключеннях цього питання не буде.

**Очікуваний результат:** Ви опинились всередині VM під користувачем `openclaw` — без жодного пароля. Запрошення буде схоже на:

```text
openclaw@devops-sandbox:~$
```

Вийдіть з VM:

```bash
exit
```

### Крок 6: Створюємо SSH-псевдонім

Набирати `openclaw@192.168.56.10` щоразу незручно. Налаштуємо псевдонім у конфігу SSH:

```bash
# Відкрийте (або створіть) файл ~/.ssh/config
# Наприклад, через nano:
nano ~/.ssh/config
```

Додайте такий блок:

```text
Host devvm-openclaw
    HostName     192.168.56.10
    User         openclaw
    IdentityFile ~/.ssh/id_ed25519
    Port         22
```

Збережіть файл (`Ctrl+O`, `Enter`, `Ctrl+X` у nano).

Встановіть правильні права:

```bash
chmod 600 ~/.ssh/config
```

Тепер перевіримо:

```bash
# Замість ssh openclaw@192.168.56.10 — просто:
ssh devvm-openclaw
```

**Очікуваний результат:** Підключення відбулось — без пароля та без введення IP-адреси. Виходьте:

```bash
exit
```

### Крок 7: Передаємо файл через SCP

Потренуємось передавати файли між хостом і VM. Створимо тестовий файл і скопіюємо його:

```bash
# Створюємо тестовий файл на хості
echo "Тестовий файл для SCP" > /tmp/test_scp.txt

# Копіюємо його на VM у домашню директорію openclaw
scp /tmp/test_scp.txt devvm-openclaw:/home/openclaw/

# Перевіряємо — заходимо на VM і дивимось чи файл там є
ssh devvm-openclaw "ls -la ~/test_scp.txt && cat ~/test_scp.txt"
```

**Очікуваний результат:** Файл з'явився на VM, виводиться його вміст.

Тепер скопіюємо файл назад — з VM на хост:

```bash
# Завантажуємо файл з VM
scp devvm-openclaw:/home/openclaw/test_scp.txt /tmp/test_scp_from_vm.txt

# Перевіряємо
cat /tmp/test_scp_from_vm.txt
```

**Очікуваний результат:** Той самий текст — файл успішно повернувся.

### Крок 8: SSH-тунель — відкриваємо прихований порт з Теми 3

Ось і головний момент цієї теми. У Темі 3 ми закінчили тим, що Python-сервер прив'язаний до `127.0.0.1:18789` — і з браузера на ноутбуці його не видно. Тепер ми виправимо це одним рядком.

**Крок 8.1.** Спочатку запустимо «симульований агент» на VM — той самий Python-сервер, що в Темі 3:

```bash
# Підключаємось як openclaw
ssh devvm-openclaw

# На VM — запускаємо сервер на прихованому порту
python3 -m http.server 18789 --bind 127.0.0.1 &

# Перевіряємо, що порт є і прив'язаний до 127.0.0.1
sudo ss -tlnp | grep 18789
# Очікуваний результат: LISTEN ... 127.0.0.1:18789 ... python3
```

**Виходимо з VM** (сервер залишається працювати у фоні):

```bash
exit
```

**Крок 8.2.** Відкриваємо SSH-тунель. Ключовий рядок:

```bash
# На хості — відкриваємо тунель:
# "Порт 8080 на моєму ноутбуці → порт 18789 на localhost VM"
ssh -L 8080:localhost:18789 devvm-openclaw
```

Термінал «завис» — це нормально. SSH тримає тунель відкритим, поки ви не закриєте цю сесію.

**Крок 8.3.** Не закриваючи цей термінал, відкрийте браузер на **вашому ноутбуці** і перейдіть за адресою:

```text
http://localhost:8080
```

**Очікуваний результат:** Сторінка відкривається — ви бачите лістинг директорії. Це файлова система VM, доступна через зашифрований тунель. Порт `18789` при цьому залишається невидимим для будь-кого крім вас.

> 💡 **Що тільки що сталось:** Запит з браузера пішов на `localhost:8080` вашого ноутбука → SSH отримав його, зашифрував → передав на VM → розпакував і надіслав до `127.0.0.1:18789` → відповідь пройшла назад тим самим шляхом. З погляду VM — хтось «локальний» звернувся до сервера. З погляду Інтернету — порту 18789 не існує.

**Крок 8.4.** Тепер закрийте термінал з SSH-тунелем (`Ctrl+C` або закрийте вікно). Спробуйте оновити сторінку в браузері.

**Очікуваний результат:** Сторінка перестала відповідати. Тунель живе тільки разом з SSH-підключенням.

**Крок 8.5.** Прибираємо за собою — зупиняємо Python-сервер:

```bash
# Підключаємось і вбиваємо фоновий процес
ssh devvm-openclaw "pkill -f 'http.server 18789'"
```

---

### Крок 9: Додаємо тунель до SSH-конфігу

Щоразу вводити `-L 8080:localhost:18789` незручно. Додамо тунель у наш SSH-конфіг як автоматичний:

```bash
nano ~/.ssh/config
```

Оновіть блок `devvm-openclaw`:

```text
Host devvm-openclaw
    HostName     192.168.56.10
    User         openclaw
    IdentityFile ~/.ssh/id_ed25519
    Port         22
    LocalForward 8080 localhost:18789
```

Тепер кожного разу, коли ви виконуєте `ssh devvm-openclaw` — тунель відкривається автоматично. Жодних додаткових прапорів.

## ✅ Результат виконання роботи

Після виконання всіх кроків у вас має бути:

- [ ] Ключова пара `~/.ssh/id_ed25519` та `~/.ssh/id_ed25519.pub` на хості
- [ ] Підключення `ssh openclaw@192.168.56.10` — без пароля
- [ ] Псевдонім `devvm-openclaw` у `~/.ssh/config` — підключення однією командою
- [ ] SCP — передача файлу на VM і назад пройшла успішно
- [ ] SSH-тунель — `http://localhost:8080` у браузері відкриває сервіс на `127.0.0.1:18789` VM

Фінальна перевірка — виконайте на хості:

```bash
# Крок 1: запускаємо сервер на VM (SSH підключається, виконує команду і виходить)
ssh devvm-openclaw "python3 -m http.server 18789 --bind 127.0.0.1 &"

# Крок 2: підключаємось — тунель відкривається автоматично через LocalForward у конфізі
ssh devvm-openclaw
```

Відкрийте `http://localhost:8080` у браузері. Якщо сторінка відкривається — завдання виконано.

> 💡 Зверніть увагу: у Кроці 2 ми не вказуємо `-L` вручну. Тунель відкрився тому, що `LocalForward` прописаний у `~/.ssh/config`. Саме це і є мета Кроку 9.
