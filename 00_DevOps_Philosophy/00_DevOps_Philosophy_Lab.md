> **Файл для студентів.** Практична частина до теорії `00_DevOps_Philosophy_Theory.md`.

## 🎯 Мета роботи

Дослідити реальний DevOps-проєкт зсередини: зрозуміти, як принципи CALMS проявляються в конкретних файлах, папках і командах реального репозиторію. Після виконання цієї роботи ви зможете читати структуру DevOps-репозиторію і розуміти, *чому* він організований саме так.

> **Що потрібно:** комп'ютер з Git та термінальним емулятором. Інтернет-доступ.

---

## 🛠 Покрокова інструкція

### Крок 1: Клонуємо репозиторій проєкту

Ми будемо досліджувати реальний проєкт AI-асистента, побудований викладачем. Він є нашим навчальним полігоном на весь курс.

```bash
# Клонуємо репозиторій у папку devops-course
git clone https://github.com/sobol-mo/devops-course.git devops-course

# Переходимо в папку проєкту
cd devops-course
```

**Очікуваний результат:** У поточній теці з'явиться папка `devops-course` із файлами проєкту.

---

### Крок 2: Досліджуємо структуру репозиторію

```bash
# Переглядаємо корінь репозиторію
ls -la

# Переглядаємо дерево папок верхнього рівня
find . -maxdepth 2 -type d | sort
```

**Очікуваний результат:** Ви побачите, що репозиторій організований за папками з числовими префіксами (`00_`, `01_`, `10_`). В одному репозиторії — і бізнес-вимоги, і архітектура, і код, і інфраструктурні скрипти. Це і є "Single Source of Truth".

---

### Крок 3: Знайдіть "Pets vs. Cattle" в реальних файлах

Відкрийте файл, де описана конфігурація сервера:

```bash
# Переглядаємо Ansible playbook для розгортання агента
cat devops-ai-assistant/10_Implementation/ansible/playbook-openclaw-setup.yml
```

**Очікуваний результат:** Ви побачите YAML-файл із інструкціями, які автоматично налаштовують сервер. Це і є Infrastructure as Code: сервер описаний як **"Cattle"** (відтворюваний із файлу), а не **"Pet"** (налаштований вручну і незамінний).

---

### Крок 4: Знайдіть принцип Least Privilege у конфігурації

```bash
# Знайдемо, де в playbook описується створення користувача
grep -A 5 "openclaw" devops-ai-assistant/10_Implementation/ansible/playbook-openclaw-setup.yml | head -40
```

**Очікуваний результат:** Ви побачите, що агент запускається від імені окремого системного користувача `openclaw` без прав `sudo`. Це реалізація принципу **Principle of Least Privilege** — навіть якщо агента зламають, зловмисник не отримає доступ до всього сервера.

---

### Крок 5: Дослідіть структуру проєкту — архітектура як код

```bash
# Переглядаємо структуру підпроєкту
ls -la devops-ai-assistant/

# Читаємо архітектурний опис системи
cat devops-ai-assistant/01_Architecture/System_Architecture.md | head -60
```

**Очікуваний результат:** Ви побачите, що поряд із кодом інфраструктури (`10_Implementation/`) знаходиться папка `01_Architecture/` з повним описом системи. Архітектура зберігається в тому ж репозиторії, що й код — це DevOps-принцип **"документація як код"** (Docs as Code): документи версіонуються, рецензуються і змінюються разом із системою, яку вони описують.

---

### Крок 6: Визначте CALMS у проєкті

Перегляньте історію змін у репозиторії:

```bash
# Переглядаємо git-лог проєкту
git log --oneline -15
```

**Очікуваний результат:** Ви побачите список комітів із повідомленнями у форматі `sync: update course materials`. Це реалізація **Sharing** (принцип S у CALMS) — кожна зміна зафіксована, датована та доступна всій команді. Зверніть також увагу на префікси повідомлень (`docs:`, `feat:`, `fix:`, `sync:`) — це **Conventional Commits**, стандарт, який робить DORA-метрику Lead Time вимірюваною.
