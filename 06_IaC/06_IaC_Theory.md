## 🎯 Бізнес-задача

Подивімось назад: ми витратили п'ять занять на ручне налаштування однієї віртуальної машини. Створили користувача, налаштували SSH-доступ, встановили Git, підготували середовище. Усе працює — але лише поки ми нічого не зламаємо.

Уявіть реальну ситуацію:

- VM зламалась через невдалий експеримент → доведеться повторити всі кроки вручну.
- Потрібно налаштувати ідентичне середовище для колеги → знову ручна робота, і немає гарантії, що результат буде однаковим.
- Потрібно розгорнути 10 серверів для Production → скільки часу це займе вручну? А якщо 100?

**Проблема:** ручне налаштування — повільне, ненадійне, неповторюване і не масштабується.

**Рішення:** описати всю конфігурацію у файлі та відтворити її автоматично за лічені хвилини.

**Зв'язок з проєктом:** У репозиторії `My_AI_Assistant` є реальний приклад такого підходу — Ansible playbook, який автоматизує повне налаштування сервера для AI-агента: створення користувача, встановлення залежностей, конфігурацію каталогів, розгортання сервісу. На занятті ми розберемо його як кейс, а в лабораторній роботі студенти напишуть аналогічну автоматизацію для свого `training-project`.

## 🧠 Необхідні знання

### 1. Pets vs. Cattle — як змінилося ставлення до серверів

Ще нещодавно в ІТ кожен сервер був як домашня тварина: мав власне ім'я, унікальне налаштування, і коли він «хворів» — адміністратор витрачав години на його «лікування». Втратити такий сервер означало катастрофу, бо ніхто не пам'ятав усіх кроків, якими його налаштовували.

Сучасний підхід — протилежний. Сервери — це «стадо» (cattle): вони однакові, замінні, і якщо один «захворів» — його знищують і створюють новий за лічені хвилини з готового опису.

| Ознака               | Pets (домашні тварини)                                | Cattle (стадо)                                                       |
| -------------------------- | ------------------------------------------------------------------- | ------------------------------------------------------------------------- |
| Ім'я                    | Унікальне:`web-prod-legacy-01`                           | Автоматичне:`node-042`                                       |
| Налаштування   | Вручну, унікальне                                    | Автоматизоване, однакове                            |
| Збій сервера    | «Лікуємо» годинами                                 | Знищуємо та створюємо заново за хвилини |
| Масштабування | Складно                                                      | Просто: запускаємо ще 10 таких самих          |
| Документація   | «Того Олексій колись налаштовував…» | Повний опис — у коді                                      |

**Аналогія з нашим курсом:** те, що ми робили в Темах 1–5 — це підхід «Pets». Кожна VM була домашньою твариною, яку ми вирощували вручну крок за кроком. Тема 6 — це перехід до підходу «Cattle»: опис конфігурації у файлі.

**Ключові поняття:**

- **Pets** — сервери з унікальним ручним налаштуванням, які «лікують» при збоях
- **Cattle** — сервери з автоматизованим налаштуванням, які замінюють при збоях

### 2. Infrastructure as Code (IaC) — що це таке

**Infrastructure as Code (IaC)** — це підхід, при якому конфігурація інфраструктури описується у файлах (коді), а не виконується вручну.

Замість того щоб для кожної нової VM вручну вводити команди `adduser`, `apt install`, `chmod`, `mkdir` — ми описуємо бажаний стан у файлі, і спеціалізований інструмент автоматично приводить сервер до цього стану.

**Що дає IaC:**

| Перевага                       | Пояснення                                                                                                        |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| **Повторюваність** | Той самий файл → той самий результат на будь-якому сервері                |
| **Версіонування**   | Файл конфігурації зберігається в Git → повна історія змін                   |
| **Масштабування**   | Налаштувати 1 сервер чи 100 — різниця в одному числі                               |
| **Документація**     | Файл конфігурації сам є документацією — видно, що і як налаштовано |
| **Аудит**                   | Хто, коли і що змінив — видно з Git-історії                                                 |

**Зв'язок з попередніми темами:** У Темі 5 ми говорили про Git як «Single Source of Truth». IaC — це продовження цієї ідеї: не тільки код програми, а й конфігурація інфраструктури живуть у Git.

**Ключові поняття:**

- **Infrastructure as Code (IaC)** — практика опису інфраструктури у файлах замість ручного налаштування

### 3. Декларативний vs. Імперативний підхід

Існують два принципово різні способи описати бажаний стан сервера.

#### Імперативний підхід: «як зробити»

Ми даємо **послідовність команд**, які треба виконати. Це звичайний скрипт:

```bash
#!/bin/bash
# Створи користувача
useradd -m appuser
# Встанови Python
apt-get install -y python3 python3-pip
# Створи директорію
mkdir -p /opt/myapp
# Зміни права
chown appuser:appuser /opt/myapp
```

**Проблема:** що станеться, якщо запустити цей скрипт двічі?

- `useradd` видасть помилку «користувач вже існує»
- `apt-get install` спробує встановити ще раз
- `mkdir` може перезаписати права

#### Декларативний підхід: «що має бути»

Ми описуємо **бажаний стан**, а інструмент сам визначає, які дії потрібні:

```yaml
- name: Створити користувача
  user:
    name: appuser
    state: present

- name: Встановити Python
  apt:
    name: python3
    state: present

- name: Створити директорію додатку
  file:
    path: /opt/myapp
    owner: appuser
    state: directory
```

**Переваги:** якщо запустити це двічі — нічого зайвого не відбудеться. Інструмент перевірить: «Користувач вже є? Так. Нічого не роблю.»

| Підхід                         | Питання            | Приклад    | Повторний запуск |
| ------------------------------------ | ------------------------- | ----------------- | ------------------------------- |
| **Імперативний**   | «Як зробити?»  | Bash-скрипт | Може зламатися     |
| **Декларативний** | «Що має бути?» | Ansible playbook  | Безпечний              |

**Ключові поняття:**

- **Імперативний підхід** — послідовність команд, що описує *як* досягти стану
- **Декларативний підхід** — опис *бажаного стану*, інструмент сам визначає кроки

### 4. Два типи IaC-інструментів: Terraform і Ansible

IaC-інструменти вирішують різні задачі і працюють на різних рівнях. Розуміння цієї різниці — ключ до правильного вибору інструменту.

#### Ви вже використовували IaC — у Темі 1

Відкрийте `Vagrantfile` з першої теми і перечитайте:

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "ubuntu/jammy64"
  config.vm.hostname = "devops-sandbox"
  config.vm.network "private_network", ip: "192.168.56.10"

  config.vm.provider "virtualbox" do |vb|
    vb.memory = 2048
    vb.cpus   = 2
  end
end
```

Це — **декларативний опис бажаного стану**: «Має існувати VM з Ubuntu 22.04, 2 GB RAM, IP 192.168.56.10». Один файл — і `vagrant up` відтворює середовище на будь-якому комп'ютері. Це і є Infrastructure as Code для локального середовища.

**Vagrant** — це provisioning-інструмент. Він відповідає на питання: **«Яка інфраструктура має існувати?»**

#### Terraform: Vagrant для хмари

**Terraform** від HashiCorp (ті самі розробники, що і Vagrant) — це те саме, але для хмарних провайдерів: AWS, Google Cloud, Hetzner, DigitalOcean.

Подивіться, наскільки схожа структура:

```hcl
# terraform/main.tf — Hetzner Cloud
resource "hcloud_server" "devops_sandbox" {
  name        = "devops-sandbox"
  server_type = "cx22"          # 2 vCPU, 4 GB RAM
  image       = "ubuntu-22.04"
  location    = "nbg1"          # Нюрнберг

  ssh_keys = [hcloud_ssh_key.my_key.id]
}
```

Якщо Vagrantfile каже «створи VM у VirtualBox з такими параметрами», то цей Terraform-код каже «створи сервер у Hetzner Cloud з такими параметрами». Різні провайдери — той самий принцип.

**Ключова риса Terraform:** після виконання `terraform apply` ви отримуєте **порожній сервер**. Операційна система встановлена, IP виданий — але більше нічого. Python не встановлений, користувача нема, конфігурації нема. Terraform зробив свою роботу: **підготував інфраструктуру**.

```
Terraform (або Vagrant)           Ansible
─────────────────────────         ──────────────────────
«Яка інфраструктура має           «Що на ній має бути
 існувати?»                        налаштовано?»

Створює VM / сервер               Налаштовує сервер
Видає IP-адресу                   Встановлює пакети
Налаштовує мережу                 Створює користувачів
                                  Копіює файли / конфіги
                                  Запускає сервіси
```

#### Чому вони зазвичай йдуть разом

У реальних DevOps-командах ці два інструменти доповнюють один одного:

1. **Terraform (або Vagrant)** — «будує будинок»: створює сервер у хмарі або локальну VM.
2. **Ansible** — «облаштовує будинок усередині»: встановлює все необхідне на вже готовий сервер.

Такий поділ відповідальності є стандартом у галузі.

#### Чому в цьому курсі ми зосереджуємося на Ansible

Terraform потребує хмарного провайдера для практики: без реального VPS `terraform apply` немає де запускати. Натомість Ansible відразу працює на VM, яку студенти вже мають з Теми 1.

У нашому курсі:

- **Provisioning** вже зроблено — VM існує завдяки Vagrant (Тема 1)
- **Configuration Management** — це наша задача зараз: налаштувати цю VM для `training-project`

Тому ми концентруємось на Ansible. Terraform залишається концептуально зрозумілим — ви вже знаєте Vagrant, а Terraform — це «Vagrant для хмари».

| Тип                      | Що робить                                  | Приклад     | Коли використовувати      |
| --------------------------- | -------------------------------------------------- | ------------------ | -------------------------------------------- |
| **Provisioning**      | Створює інфраструктуру        | Terraform, Vagrant | Потрібен новий сервер     |
| **Config Management** | Налаштовує існуючий сервер | Ansible            | Сервер є, треба готувати |

**Ключові поняття:**

- **Provisioning** — створення інфраструктури (серверів, мереж) з нуля; приклади: Terraform, Vagrant
- **Configuration Management** — налаштування існуючих серверів до бажаного стану; приклад: Ansible
- **Terraform** — HashiCorp-інструмент для опису хмарної інфраструктури у декларативному форматі (HCL)
- **HCL (HashiCorp Configuration Language)** — мова опису конфігурації, яку використовує Terraform

### 5. Ansible: основні концепції

#### Чому Ansible?

Порівняно з іншими Configuration Management інструментами (Chef, Puppet), Ansible має кілька ключових переваг для початківців:

- **Agentless** — не потребує встановлення додаткового ПЗ на сервері. Працює через звичайний SSH, який ми вже налаштували в Темі 4.
- **YAML** — конфігурація пишеться в зрозумілому форматі, а не в спеціальній мові програмування.
- **Push-модель** — ви запускаєте Ansible на своєму комп'ютері, і він «штовхає» конфігурацію на сервер через SSH.

```
┌───────────────────┐        SSH        ┌───────────────────┐
│  Ваш комп'ютер    │ ──────────────▶  │  Цільовий сервер  │
│  (Control Node)   │                   │  (Managed Node)   │
│                   │                   │                   │
│  ansible-playbook │                   │  Не потребує      │
│  playbook.yml     │                   │  агента — тільки  │
│  inventory.ini    │                   │  SSH + Python      │
└───────────────────┘                   └───────────────────┘
```

#### Анатомія Ansible-проєкту

Мінімальний Ansible-проєкт складається з трьох файлів:

```
ansible/
├── ansible.cfg       ← Налаштування Ansible (де шукати inventory, тощо)
├── inventory.ini     ← Список серверів, на яких виконувати команди
└── playbook.yml      ← Опис бажаного стану (що саме налаштувати)
```

#### Inventory — хто є хто

Inventory (інвентар) — файл, який повідомляє Ansible, на яких серверах працювати:

```ini
[devservers]
devvm ansible_host=192.168.56.10 ansible_user=vagrant
```

- `[devservers]` — група серверів (можна мати кілька груп)
- `devvm` — ім'я хоста (для зручності)
- `ansible_host` — IP-адреса сервера
- `ansible_user` — користувач для SSH-підключення

#### Playbook — що робити

Playbook (плейбук) — файл з описом бажаного стану. Це серце Ansible:

```yaml
---
- name: Налаштування Dev-сервера
  hosts: devservers
  become: yes

  tasks:
    - name: Створити користувача для додатку
      user:
        name: appuser
        shell: /bin/bash
        state: present

    - name: Встановити необхідні пакети
      apt:
        name:
          - python3
          - python3-pip
          - python3-venv
        state: present
        update_cache: yes
```

**Розберемо кожен елемент:**

| Елемент | Що означає                                                                                            |
| -------------- | -------------------------------------------------------------------------------------------------------------- |
| `name`       | Людиночитабельний опис дії — виводиться при запуску               |
| `hosts`      | Для яких серверів (група з inventory)                                                     |
| `become`     | Виконувати від імені `root` (аналог `sudo`)                                        |
| `tasks`      | Список завдань — виконуються послідовно                                     |
| `user`       | Модуль Ansible для управління користувачами                                    |
| `apt`        | Модуль для управління пакетами (Ubuntu/Debian)                                      |
| `state`      | Бажаний стан:`present` = має існувати, `absent` = має бути видалений |

#### Модулі — будівельні блоки

Кожна задача (task) в Ansible використовує **модуль** — готовий компонент для конкретної операції:

| Модуль | Призначення                                            | Приклад                                             |
| ------------ | ----------------------------------------------------------------- | ---------------------------------------------------------- |
| `user`     | Управління користувачами                   | Створити/видалити користувача   |
| `apt`      | Управління пакетами                             | Встановити Python, Git                           |
| `file`     | Управління файлами та директоріями | Створити каталог, змінити права |
| `copy`     | Копіювання файлів на сервер               | Скопіювати конфігурацію              |
| `template` | Копіювання файлів зі змінними           | Згенерувати конфігурацію            |
| `pip`      | Управління Python-пакетами                      | Встановити Flask                                 |
| `systemd`  | Управління сервісами                           | Запустити/зупинити сервіс           |

Ansible має **сотні** вбудованих модулів. Ми використаємо лише декілька — цього достатньо для налаштування сервера.

**Ключові поняття:**

- **Control Node** — комп'ютер, з якого запускається Ansible
- **Managed Node** — сервер, який Ansible налаштовує
- **Inventory** — файл зі списком серверів
- **Playbook** — файл з описом бажаного стану серверів
- **Task** — окреме завдання в playbook
- **Module** — готовий компонент для виконання конкретної операції (user, apt, file, copy та інші)

### 6. Ідемпотентність — чому це критично

**Ідемпотентність** — це властивість операції давати однаковий результат незалежно від кількості запусків.

Це ключове поняття в IaC. Якщо playbook ідемпотентний — його можна запускати 1 раз, 5 разів, 100 разів, і результат буде однаковим: сервер буде у бажаному стані.

**Приклад:** задача «Створити користувача appuser»

| Запуск | Стан до запуску           | Що робить Ansible      | Результат |
| ------------ | -------------------------------------- | ------------------------------ | ------------------ |
| 1-й         | Користувача немає      | Створює `appuser`     | `changed`        |
| 2-й         | Користувач вже існує | Нічого не робить | `ok`             |
| 3-й         | Користувач вже існує | Нічого не робить | `ok`             |

**Порівняння з bash-скриптом:**

```bash
# Bash — НЕ ідемпотентний
useradd appuser    # 1-й раз: ОК
useradd appuser    # 2-й раз: ПОМИЛКА — користувач вже існує!
```

```yaml
# Ansible — ідемпотентний
- name: Створити користувача
  user:
    name: appuser
    state: present   # "Має існувати" — Ansible сам перевірить
```

**Вивід Ansible при повторному запуску** — зверніть увагу на `changed=0`:

```
PLAY RECAP *********************************************************************
devvm : ok=5    changed=0    unreachable=0    failed=0    skipped=0
```

Це означає: «Я перевірив усі 5 завдань — сервер вже у бажаному стані, нічого не змінював».

**Чому це важливо:**

- Безпечно запускати playbook повторно (наприклад, після додавання нової задачі)
- Можна використовувати в автоматизації (CI/CD) без побоювань зламати сервер
- Гарантує, що стан сервера відповідає опису в коді

**Ключові поняття:**

- **Ідемпотентність** — властивість операції давати однаковий результат при повторному виконанні
- **changed** — Ansible повідомляє, що щось змінив на сервері
- **ok** — Ansible перевірив і нічого не змінював — стан вже відповідає опису

### 7. Реальний приклад: як виглядає IaC у продакшн-проєкті

Розглянемо як **кейс** реальний Ansible playbook із репозиторію `My_AI_Assistant`. Цей playbook повністю автоматизує налаштування Production-сервера для AI-агента.

> **Примітка:** Цей приклад показано для ілюстрації масштабу та структури реального IaC. На лабораторній студенти напишуть аналогічний, але спрощений playbook для свого `training-project`.

#### Структура реального playbook

Playbook складається з логічних секцій, кожна з яких відповідає за окрему частину налаштування:

```
playbook-openclaw-setup.yml
├── vars:                     ← Змінні (імена, шляхи)
├── tasks:
│   ├── User Creation         ← Створення системного користувача
│   ├── Node.js Installation  ← Встановлення runtime-залежностей
│   ├── Directory Setup       ← Створення каталогів з правильними правами
│   ├── Application Install   ← Встановлення додатку
│   ├── PATH Configuration    ← Налаштування оточення
│   ├── Systemd Service       ← Налаштування автозапуску
│   └── Final Status          ← Підсумок
```

#### Фрагмент: створення користувача

```yaml
vars:
  app_user: openclaw
  app_home: /home/openclaw
  app_dir: /opt/openclaw

tasks:
  - name: Create application user
    user:
      name: "{{ app_user }}"
      shell: /bin/bash
      create_home: yes
      groups: []  # Мінімальні права — Principle of Least Privilege
      state: present
```

Зверніть увагу:

- **`{{ app_user }}`** — змінна, оголошена в секції `vars`. Змінювати ім'я користувача потрібно в одному місці.
- **`groups: []`** — порожній список груп. Це той самий принцип найменших привілеїв, який ми вивчали в Темі 2.
- **`state: present`** — декларативний опис: «цей користувач має існувати».

#### Фрагмент: ідемпотентне встановлення залежностей

```yaml
  - name: Check if runtime is installed
    command: node --version
    register: runtime_check
    ignore_errors: yes
    changed_when: false

  - name: Install runtime
    apt:
      name: nodejs
      state: present
    when: runtime_check.rc != 0
```

Тут видно два важливі патерни:

- **`register`** — зберігає результат команди в змінну для подальших перевірок
- **`when`** — умовне виконання: встановлюємо тільки якщо ще не встановлено
- **`changed_when: false`** — повідомляє Ansible, що команда `--version` не змінює стан сервера

Це забезпечує ідемпотентність: при повторному запуску Ansible бачить, що runtime вже є, і пропускає встановлення.

#### Що ми побачимо при запуску

```bash
ansible-playbook -i inventory.ini playbook.yml
```

```
PLAY [Налаштування сервера] ****************************************************

TASK [Create application user] *************************************************
changed: [devvm]

TASK [Install Python packages] *************************************************
changed: [devvm]

TASK [Create application directory] ********************************************
changed: [devvm]

PLAY RECAP *********************************************************************
devvm : ok=8    changed=5    unreachable=0    failed=0
```

Другий запуск — і результат інший:

```
PLAY RECAP *********************************************************************
devvm : ok=8    changed=0    unreachable=0    failed=0
```

`changed=0` — сервер уже у бажаному стані. Ідемпотентність у дії.
