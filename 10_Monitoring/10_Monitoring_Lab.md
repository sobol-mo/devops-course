## 🗺 Де ми зараз? Підсумок пройденого шляху

Всі попередні лабораторні роботи (Теми 6–9) були присвячені одному великому процесу: **Delivery** (доставці коду на сервер). Ми створили сервер (Vagrant), налаштували його (Ansible), запустили сервіс (systemd), підняли базу (Docker) і автоматизували оновлення через CI/CD.

Тепер наш код успішно живе в "production". Але робота DevOps-інженера на цьому не закінчується. CI/CD каже, що код доїхав. А хто скаже, чи він досі працює? 

У цій лабораторній роботі ми переходимо до етапу **Operation та Monitoring**. Ми створимо просту, але дієву систему контролю стабільності для нашого `training-project`. Це мініатюрна версія того, що в реальному житті роблять інструменти на зразок Prometheus.

## 🎯 Мета роботи

Налаштувати базовий механізм моніторингу стабільності сервісу. 
Після виконання роботи ви матимете скрипт, який регулярно опитує Flask-додаток, веде історію перевірок (Logging), генерує Alert із захистом від мікро-збоїв, та працює автоматично у фоновому режимі (через `cron`). Також ви навчитеся використовувати `journalctl` для розуміння причин збою (Observability).

## 🛠 Покрокова інструкція

### Крок 1: Перевірка передумов

Спочатку переконаємось, що система-пацієнт (наша віртуальна машина) жива.

На **хості** (вашому ноутбуці):
```bash
# Переходимо в папку проєкту
cd ~/devops-course/training-project/

# Перевіряємо статус ВМ
vagrant status

# Якщо вона зупинена, піднімаємо:
vagrant up

# Перевіряємо, що health-endpoint нашого сервісу відповідає
ssh vagrant@192.168.56.10 "curl -s http://localhost:5000/health"
```

**Очікуваний результат:** команда `curl` має повернути `{"status": "ok"}`. Це означає, що сервіс готовий до моніторингу.

---

### Крок 2: Створення базового скрипта моніторингу

Зайдіть на віртуальну машину:
```bash
ssh vagrant@192.168.56.10
```

Створимо скрипт `monitor.sh` у домашній директорії:
```bash
nano ~/monitor.sh
```

Додайте в нього наступний код:
```bash
#!/bin/bash

# Налаштування
URL="http://localhost:5000/health"
LOG_FILE="/home/vagrant/monitor.log"
TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')

# Виконуємо HTTP-запит і витягуємо тільки статус-код (наприклад, 200 або 500)
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" "$URL")

# Логіка перевірки
if [ "$HTTP_CODE" == "200" ]; then
    echo "[$TIMESTAMP] INFO: Service is healthy (HTTP 200)" >> "$LOG_FILE"
else
    echo "[$TIMESTAMP] ERROR: Service returned HTTP $HTTP_CODE" >> "$LOG_FILE"
fi
```

Збережіть файл (`Ctrl+O`, `Enter`, `Ctrl+X`) і зробіть його виконуваним:
```bash
chmod +x ~/monitor.sh
```

Перевірте, як він працює:
```bash
./monitor.sh
cat ~/monitor.log
```

**Очікуваний результат:** у файлі логів з'явиться рядок `[Дата Час] INFO: Service is healthy (HTTP 200)`.

### Крок 3: Додаємо захист від Alert Fatigue (поріг збоїв)

Як ми обговорювали в теорії, будити інженера через один випадковий збій мережі не можна. Наш моніторинг має підняти тривогу тільки якщо сервіс лежить **стабільно** (наприклад, 3 перевірки поспіль).

Відкрийте скрипт знову:
```bash
nano ~/monitor.sh
```

І повністю замініть його вміст на цей:
```bash
#!/bin/bash

URL="http://localhost:5000/health"
LOG_FILE="/home/vagrant/monitor.log"
STATE_FILE="/tmp/monitor_failures.txt"
TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')

# Якщо файлу стану ще немає, створюємо його з нулем
if [ ! -f "$STATE_FILE" ]; then
    echo "0" > "$STATE_FILE"
fi

# Читаємо поточну кількість помилок поспіль
FAILURES=$(cat "$STATE_FILE")

HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" "$URL")

if [ "$HTTP_CODE" == "200" ]; then
    echo "[$TIMESTAMP] INFO: Service is healthy (HTTP 200)" >> "$LOG_FILE"
    # Скидаємо лічильник помилок після успішної перевірки
    echo "0" > "$STATE_FILE"
else
    echo "[$TIMESTAMP] ERROR: Service returned HTTP $HTTP_CODE" >> "$LOG_FILE"
    
    # Збільшуємо лічильник
    FAILURES=$((FAILURES+1))
    echo "$FAILURES" > "$STATE_FILE"
    
    # Перевіряємо, чи досягнуто поріг алертів (3 рази)
    if [ "$FAILURES" -ge 3 ]; then
        echo "[$TIMESTAMP] 🚨 ALERT: Service failed $FAILURES times in a row! Requires investigation." >> "$LOG_FILE"
        # У реальному житті тут був би виклик curl для відправки повідомлення в Telegram чи Slack
    fi
fi
```

Тепер наш моніторинг "розумний" — він прощає поодинокі помилки і починає "кричати" лише коли сервіс дійсно лежить.

### Крок 4: Автоматизація перевірок (Cron)

Щоб скрипт став справжньою системою моніторингу, він має запускатися сам. У Linux найпростіший планувальник задач — це `cron`.

На VM виконайте команду:
```bash
crontab -e
```
*(Якщо система запитає, який редактор використати, оберіть `nano`, зазвичай це варіант 1).*

Додайте в самий кінець файлу наступний рядок:
```bash
* * * * * /home/vagrant/monitor.sh
```
Цей синтаксис означає: "запускати скрипт кожну хвилину, кожної години, кожного дня".

Збережіть файл. Ви побачите повідомлення: `crontab: installing new crontab`.
Тепер `cron` кожну хвилину непомітно опитуватиме ваш сервіс.

### Крок 5: Імітація аварії (Chaos Engineering)

Час перевірити, чи спрацює наш моніторинг у бойових умовах. Ми свідомо "вб'ємо" сервіс.

На VM відкрийте другий термінал, або в цьому ж терміналі почніть безперервно читати лог:
```bash
tail -f ~/monitor.log
```
Ви побачите, як щохвилини додається новий рядок `INFO: Service is healthy`.

Не зупиняючи `tail` (або в сусідньому вікні терміналу), зупиніть Flask-додаток:
```bash
sudo systemctl stop training-app
```

Спостерігайте за `monitor.log`:
1. На першій хвилині ви побачите помилку (HTTP 000 — бо порт взагалі не відповідає).
2. На другій хвилині — друга помилка.
3. На третій хвилині вискочить **🚨 ALERT**. 

Моніторинг працює! Він помітив тривалу зупинку.

### Крок 6: Від Моніторингу до Observability (`journalctl`)

Отже, моніторинг повідомив нам "Сервіс не відповідає". Але він не каже **ЧОМУ**.
Для цього потрібна **Observability** (спостережуваність), а саме — вивчення логів самого сервісу.

У нас сервіс працює під управлінням `systemd` (з Теми 7). У `systemd` є вбудована система логування — `journald`.

Подивимось логи нашого сервісу:
```bash
# Переглядаємо останні події нашого застосунку
sudo journalctl -u training-app --no-pager | tail -n 20
```

Ви маєте побачити записи про те, що сервіс отримав сигнал `SIGTERM` і коректно зупинився.

Запустіть сервіс назад:
```bash
sudo systemctl start training-app
```

Подивіться логи моніторингу:
```bash
tail -n 5 ~/monitor.log
```
Очікувано, наступна хвилина принесе `INFO: Service is healthy`. Інцидент вичерпано.
